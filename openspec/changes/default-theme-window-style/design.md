## Context

当前 `Window` 组件已具备完整交互能力（激活、拖拽、缩放、控制按钮），但 Default 主题下视觉层级较弱，难以快速识别“窗口边界、标题栏、内容区”三段结构。需求强调“越简单越好”，并明确禁止阴影，因此需要通过边框、背景对比和轻量状态差异来建立可辨识性。

该变更应保持现有组件行为、事件回调和公开 API 完全不变，仅调整默认样式表达；同时保证样式实现延续项目现有 SCSS 与类名约定，便于后续主题扩展。

## Goals / Non-Goals

**Goals:**
- 在 Default 主题下提供 Window 的最小可辨识视觉方案：清晰边框、可区分标题栏、明确内容区。
- 不使用阴影，仍能让窗口在页面中被直观看作独立容器。
- 保持 `Window` 组件交互与 API 不变，避免引入行为回归。
- 使样式可测试（通过类名与样式断言验证关键视觉约束）。

**Non-Goals:**
- 不重做主题系统或新增主题 token 体系。
- 不新增窗口动画、毛玻璃、阴影等装饰效果。
- 不变更拖拽/缩放/激活逻辑与现有事件语义。
- 不引入新的外部依赖。

## Decisions

1) 使用“边框 + 标题栏对比 + 内容区背景”建立窗口识别
- Decision: 在 `Window.scss` 中优先使用 1px 边框、标题栏与主体轻度色差，而非视觉特效。
- Why: 这是满足“简单且可辨识”的最低复杂度方案，也天然符合“不要阴影”的硬约束。
- Alternatives considered:
  - 增强外阴影：识别度高，但与需求冲突（禁止阴影）。
  - 仅改标题文字样式：改动最小，但容器边界仍不明显，识别不足。

2) 活动态与非活动态采用轻量差异而非大幅视觉跳变
- Decision: `cm-window` 与 `cm-window--inactive` 保持同一结构，仅在边框/标题栏色值做小幅差异。
- Why: 避免视觉噪声，保持简洁，同时保留“焦点窗口”可感知性。
- Alternatives considered:
  - 非活态大幅降透明度：虽明显，但影响内容可读性，不够稳健。
  - 非活态不区分：最简单，但会降低多窗口场景可用性。

3) 样式改动集中在现有 Window 结构，尽量不改 TSX 结构
- Decision: 首选仅更新 `Window.scss`；仅在测试可观测性不足时，才在 `Window.tsx` 补充语义 className。
- Why: 最小化回归风险，遵循“改视觉不改行为”的边界。
- Alternatives considered:
  - 重构 DOM 结构以支持更多皮肤层：灵活但超出本次范围。

4) 通过测试锁定关键视觉约束
- Decision: 在 `tests/Window.test.tsx` 补充断言，覆盖“存在窗口容器结构类名/状态类名”和“无阴影约束（不依赖视觉快照）”。
- Why: 用稳定、低脆弱度的断言验证样式契约，避免仅靠人工预览。
- Alternatives considered:
  - 引入截图比对：准确但成本更高、维护复杂，不适合本次简单改动。

## Risks / Trade-offs

- [Risk] 无阴影条件下，某些背景场景中窗口与页面分离度仍可能不足  
  → Mitigation: 提高边框与标题栏的对比度，并保持内容区与标题栏的结构差异。

- [Risk] 过于保守的改动可能导致“变化不明显”  
  → Mitigation: 在测试与开发预览中确认“标题栏、边框、非活态差异”三项都可感知。

- [Risk] 增加样式断言后，未来主题微调可能触发测试更新  
  → Mitigation: 断言聚焦结构与约束（如无阴影、状态类存在），避免绑定脆弱的具体像素值。
