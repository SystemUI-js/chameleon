## Context

当前组件库已经提供 `CWindow` 与窗口头部、内容区域等基础结构，但缺少与窗口场景配套的状态栏抽象。提案要求补充独立的 `CStatusBar` 与 `CStatusBarItem`，并保持它们与 `CWindow` 解耦，使消费方既能在窗口底部使用，也能在其他布局中复用。

项目本身是 React + TypeScript 组件库，约束包括：

- 组件应延续现有 `src/components/` 的组织方式与命名模式。
- 样式采用独立 SCSS/CSS 文件与 className 拼接，不引入 CSS-in-JS。
- 对外导出统一从 `src/index.ts` 暴露，测试通过 Jest + React Testing Library 验证。

该变更影响组件实现、样式命名、公开导出与测试覆盖，但不涉及状态管理、数据模型或运行时依赖变更。

## Goals / Non-Goals

**Goals:**
- 提供独立的 `CStatusBar` 容器组件，默认支持横向排列与基础样式钩子。
- 提供独立的 `CStatusBarItem` 单元组件，用于承载状态信息并支持灵活组合。
- 保持与 `CWindow` 的组合关系而非内置耦合，避免改变现有窗口 API。
- 补齐导出、基础渲染测试和组合使用验证，确保组件可被消费方直接使用。

**Non-Goals:**
- 不为 `CWindow` 新增 `statusBar` 插槽、专用属性或自动挂载逻辑。
- 不引入状态栏专属行为能力，例如自动分隔、溢出处理、响应式折叠或交互菜单。
- 不扩展主题系统或新增复杂 token，仅复用现有样式组织方式完成基础外观。

## Decisions

### 1. 采用两个独立组件，而不是把状态栏嵌入 `CWindow`

`CStatusBar` 负责容器语义与布局，`CStatusBarItem` 负责单元包装。这样可以与提案一致，确保状态栏可复用于窗口之外，同时避免为了单一场景扩张 `CWindow` 的职责。

备选方案：
- 在 `CWindow` 内置状态栏区域：会让窗口组件承担更多布局职责，削弱组合弹性。
- 只提供 `CStatusBar`，不提供 `CStatusBarItem`：会减少最小抽象，导致消费方难以复用统一的 item 样式钩子。

### 2. 延续现有组件库模式，提供轻量包装和 `className` 扩展能力

两个组件都应保持“样式壳子”定位，仅暴露符合现有组件风格的基础 DOM 结构、默认类名与 `className` 合并能力，不耦合业务数据或上下文。这样能与当前组件库的低约束设计保持一致。

备选方案：
- 增加上下文共享或注册机制：当前需求没有复杂交互，额外抽象会提高维护成本。
- 强制子组件只能挂在 `CStatusBar` 内：与提案中“允许按需组合到任意位置”冲突。

### 3. 样式实现以横向 flex 为核心，保留最小默认视觉

`CStatusBar` 默认使用横向 `flex` 布局对齐 item，`CStatusBarItem` 提供稳定的 class 钩子和基础内边距/对齐方式即可。组件不承担复杂视觉规范，把定制空间留给使用方和后续主题演进。

备选方案：
- 增加大量视觉 token 或变体：当前收益不足，且会放大设计面。
- 完全无默认样式：虽然更轻，但会降低组件作为“状态栏”抽象的可识别性，不利于示例与测试。

### 4. 公开导出、示例与测试作为同一交付面完成

该变更的可用性不仅取决于组件代码，还取决于消费方能否从入口导入，以及开发预览/测试是否能证明组件与 `CWindow` 的组合关系。因此导出更新、渲染测试与组合示例需要与组件实现同步完成。

备选方案：
- 仅新增组件文件，不更新入口或测试：会导致功能处于“存在但不可验证”的状态，无法满足交付要求。

## Risks / Trade-offs

- [状态栏样式过轻，难以体现组件价值] → 通过提供稳定类名和基础布局，先满足结构复用；更强视觉规范留给后续主题演进。
- [与现有窗口样式组合时出现边框或间距不一致] → 在开发预览和测试中加入 `CWindow` + `CStatusBar` 组合示例，尽早校验默认表现。
- [组件 API 过早复杂化] → 当前仅保留基础容器属性与 `className` 扩展，不提前加入未经验证的变体能力。
- [缺少导出或测试导致消费方无法使用] → 将入口导出与测试补齐视为本次实现的必要交付，而非后置工作。

## Migration Plan

1. 在 `src/components/` 中新增 `CStatusBar`、`CStatusBarItem` 及对应样式文件。
2. 更新组件库入口导出，确保消费方可直接导入新组件。
3. 补充测试，覆盖基础渲染、子项组合与公开导出。
4. 更新开发预览或示例，验证与 `CWindow` 的组合效果。
5. 若默认样式引发兼容问题，可通过回滚导出与组件文件撤回该能力；因未修改既有 `CWindow` API，回滚风险较低。

## Open Questions

- `CStatusBarItem` 是否需要默认分隔线、自动占位等更强视觉约定？当前设计选择不包含，待真实使用场景验证后再决定。
- 开发预览中是否已有适合展示窗口组合的示例入口？若没有，需在实现阶段选择最贴近现有演示结构的方式补充。
