## Context

当前 `CMenu` 仅维护一个根级 `trigger` 配置，并在递归渲染子菜单时通过 `parentTrigger` 向下继承默认触发方式。这个实现会让根菜单使用 `click` 打开时，所有嵌套子菜单也默认只能通过 click 展开，和桌面应用常见的“点击打开主菜单、悬停切换子菜单”交互不一致。

现有实现已经具备以下可复用基础：

- 根菜单开关由 `isRootOpen` 控制，分支展开由 `openBranchByDepth` 控制；
- 菜单项已经支持 `item.trigger` 显式覆盖；
- 父级菜单项同时具备 `onClick` 与 `onPointerEnter` 入口，只是当前 hover 是否生效取决于继承到的触发模式；
- 测试与开发夹具已经覆盖纯 click / 纯 hover 模式，可在此基础上补充混合模式验证。

本次变更需要在尽量不扩大公开 API 面的前提下，重新定义“默认触发方式”的解析规则：根菜单仍由根级 `trigger` 决定，而进入菜单树后，嵌套子菜单应按更符合桌面菜单心智的默认行为展开，同时保留每个菜单项的显式覆盖能力。

## Goals / Non-Goals

**Goals:**

- 保持根菜单默认通过 `click` 打开，不破坏现有根触发方式配置。
- 让根菜单以 `click` 打开时，嵌套子菜单默认通过 `hover` 展开。
- 保持 `MenuListItem.trigger` 的优先级最高，允许单个父菜单项显式改回 `click` 或继续使用 `hover`。
- 尽量复用现有状态结构与递归渲染逻辑，避免引入新的复杂状态机。
- 为混合触发模式补充单元测试与开发示例，确保行为可验证、可演示。

**Non-Goals:**

- 不引入新的外部依赖。
- 不在本次变更中实现键盘方向键导航、焦点漫游或延迟开关策略。
- 不新增独立的公开 props（如 `submenuTrigger`）来暴露更多配置维度，除非实现过程中证明现有语义无法覆盖需求。
- 不调整叶子节点选择、外部点击关闭、主题样式等与本次触发语义无关的行为。

## Decisions

### 1. 将“默认触发方式”改为深度感知的解析规则

决策：保留根级 `trigger` 作为根菜单打开方式；在递归渲染子菜单时，不再简单继承父级默认值，而是引入“根级默认 + 嵌套默认”解析规则：

- 深度 `0`（根触发器）仍使用 `trigger ?? 'click'`；
- 深度 `1+` 的父菜单项，如果没有显式 `item.trigger`，则在根菜单为 `click` 模式时默认使用 `hover`；
- 当根菜单本身为 `hover` 模式时，嵌套子菜单继续默认使用 `hover`，保持现有 hover 菜单体验一致。

原因：这能直接满足“click 打开总菜单、hover 展开子菜单”的核心诉求，同时不需要用户理解额外的新 API。

备选方案：

- 继续沿用单一继承规则：实现最简单，但无法满足本次交互目标。
- 新增公开属性如 `submenuTrigger`：灵活性更高，但会扩大 API 面、增加文档与兼容成本；在当前需求下属于过度设计。

### 2. 保留 `item.trigger` 显式覆盖，并使其优先于深度默认值

决策：对子菜单父项的触发方式统一通过解析函数计算，优先级为 `item.trigger` > 深度默认值。

原因：现有 `MenuListItem` 已经提供逐项定制能力，本次变更应保留这一能力，避免破坏已有配置，也方便为特殊分支保留 click 展开语义。

备选方案：

- 让深度默认值覆盖 `item.trigger`：会破坏现有可配置能力，与 proposal 中“显式触发配置仍可覆盖默认行为”的目标冲突。
- 仅在根节点使用混合模式，子节点完全忽略显式配置：规则更少，但扩展性不足，且不兼容现有数据结构设计。

### 3. 延续现有状态模型，仅调整父节点事件触发条件

决策：继续使用 `isRootOpen` 与 `openBranchByDepth` 管理打开状态，不新增额外状态树；实现重点放在父菜单项默认触发模式的解析，以及 `onPointerEnter` / `onClick` 在不同触发模式下的执行条件。

具体上：

- `hover` 作为子菜单默认模式时，`onPointerEnter` 负责“主动打开/切换”分支；
- `onClick` 仍保留为父项交互入口，用于显式 `click` 模式以及 hover 模式下的兼容性兜底，不改变叶子项选择后关闭菜单的既有行为；
- `openBranchByDepth` 继续保证同一深度仅保留一个打开分支，避免同时展开多个兄弟子菜单。

原因：当前状态结构已足以表达“根已开、每层一个活动分支”的菜单模型，改动成本低，风险可控。

备选方案：

- 引入完整菜单树状态或有限状态机：可扩展到更复杂交互，但对当前需求明显过重。
- 为 hover 模式单独维护悬停状态：会增加同步与清理复杂度，而现有分支状态已能满足渲染需要。

### 4. 通过测试与开发夹具显式覆盖混合模式场景

决策：在 `tests/Menu.test.tsx` 中新增混合触发场景断言，并在 `src/dev/playwright/menuHarness.tsx` / `src/dev/ComponentCatalog.tsx` 增加可演示的 mixed fixture 或示例。

至少覆盖：

- 根触发器 click 打开总菜单；
- 未显式配置触发方式的父级子菜单在 pointer enter 时展开；
- 显式 `trigger: 'click'` 的子菜单项仍需 click 才展开；
- 叶子节点选择、外部点击关闭等既有行为在混合模式下保持正常。

原因：该变更的价值主要体现在交互语义上，仅靠静态代码难以防止回归；测试与演示需要同步更新。

备选方案：

- 只修改实现不补测试：交互回归风险高。
- 只补测试不加开发示例：本地验证与后续文档演示成本较高。

## Risks / Trade-offs

- **[风险] 触发规则从“单一继承”变为“深度感知”，理解成本上升** → **缓解：** 在组件注释、示例与后续 spec 中明确优先级：根级 `trigger` 控制根菜单，`item.trigger` 覆盖子菜单默认值，未显式配置时嵌套子菜单遵循混合默认规则。
- **[风险] hover 子菜单与 click 兜底并存，可能出现比预期更宽松的交互** → **缓解：** 在测试中明确哪些场景必须由 hover 主动展开，哪些场景允许 click 兼容；实现时避免让 hover 默认分支失去显式 click 覆盖能力。
- **[风险] 深度默认值切换可能影响现有依赖“根 click → 子项 click”旧行为的调用方** → **缓解：** 保留 `item.trigger='click'` 覆盖路径，并在 changelog / spec 中注明兼容迁移方式。
- **[风险] 开发演示与测试夹具未同步更新，导致行为被误判** → **缓解：** 将 mixed fixture 纳入开发预览入口，并补充自动化测试覆盖关键路径。

## Migration Plan

1. 调整 `Menu.tsx` 的触发方式解析逻辑，抽离根级与嵌套级默认规则。
2. 补充单元测试，验证 mixed 模式及显式覆盖场景。
3. 更新开发夹具/组件目录示例，便于人工回归确认。
4. 在 `[Unreleased]` 中记录该交互语义变更与显式覆盖迁移方式。

回滚策略：如果 mixed 模式引发不可接受的兼容问题，可回退到当前单一继承实现；因为本次不引入数据结构迁移，也不涉及持久化状态，回滚成本较低。

## Open Questions

- hover 默认的父级菜单项在 click 时是否应继续切换展开状态，还是只作为无障碍/触屏兼容入口保留？实现前需要结合现有测试期望最终确认。
- 组件目录中是新增独立“mixed”示例，还是在现有 menu 示例中直接展示混合默认行为，更利于用户理解？
