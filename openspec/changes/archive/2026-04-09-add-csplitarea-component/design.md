## Context

当前组件库已经具备 `Window`、`Dock`、`Grid` 等基础布局与容器能力，但缺少一个面向 IDE / 工作区场景的可嵌套区域分割容器。提案要求新增 `CSplitArea`，支持横向与纵向分割、通过多个 `children` 自动生成区域、在子区域中继续嵌套使用，并在开启 `separatorMovable` 时提供拖动分割线交互。

项目约束包括：

- 保持 React 18 + TypeScript 严格类型风格，不引入 `any`
- 组件样式继续采用独立 SCSS 文件与约定式 className
- 组件需接入现有导出链路，并补充开发示例与测试
- `@system-ui-js/multi-drag` 已在项目依赖中，可直接作为拖动能力基础设施使用

## Goals / Non-Goals

**Goals:**

- 提供一个可复用的 `CSplitArea` 组件，基于子元素数量自动渲染区域与分割线
- 支持 `horizontal` / `vertical` 两种方向，并允许区域内部继续嵌套 `CSplitArea`
- 在 `separatorMovable` 启用时，让相邻区域尺寸可通过拖拽调整，且在子区域移除后重新分配布局
- 为开发预览提供横竖嵌套、动态销毁子区域的演示，验证组件行为与 API 可用性

**Non-Goals:**

- 本次不实现布局持久化、受控尺寸模型或外部状态同步协议
- 本次不扩展复杂约束能力，如区域折叠、吸附、最小化动画或多分隔条联动策略
- 本次不引入新的布局引擎，继续基于现有 React 渲染与 CSS 布局机制实现

## Decisions

### 1. 使用 `children` 归一化结果作为区域数据源

`CSplitArea` 通过 `React.Children.toArray` 过滤有效子节点，并以此生成区域列表与分割线，而不是要求调用方额外提供面板配置对象。这样可以保持 API 与 React 组合模式一致，天然支持嵌套写法，也更符合组件库当前的使用习惯。

备选方案：使用 `items` 配置数组描述每个区域。该方案更适合复杂受控布局，但会增加调用成本，并削弱 JSX 组合能力，因此本次不采用。

### 2. 使用 Flex 布局承载方向切换与区域伸缩

组件根容器使用单层 flex 布局，根据 `direction` 切换主轴方向；每个子区域使用独立包装元素承接内容与尺寸控制。这样可以最小化 DOM 复杂度，便于横竖方向复用同一套渲染逻辑，也便于后续为分割线样式和命中区域预留扩展空间。

备选方案：使用 CSS Grid 管理轨道尺寸。Grid 更适合静态网格，但在动态插入分割线、维护相邻轨道拖动关系时心智负担更高，因此优先采用 flex。

### 3. 用内部比例状态管理区域尺寸，而不是直接写死像素

组件内部维护与区域数量对应的比例数组，初始均分；拖动分割线时只调整相邻两个区域的比例。这样可以在容器尺寸变化时保持相对布局稳定，也能在子区域动态增减时按当前可见区域数量重新归一化，避免像素值在不同容器尺寸下失真。

备选方案：直接保存像素宽高。像素值实现简单，但对响应式容器和嵌套场景适应性较差，因此不采用。

### 4. 将 `@system-ui-js/multi-drag` 封装在分割线层，而不是暴露到底层 API

拖动能力只作为 `separatorMovable` 打开时的内部实现细节：组件为每个分割线绑定对应 ref 和拖动生命周期，在拖动回调中根据方向读取位移并更新相邻比例。对外仍只暴露声明式属性，不把第三方库实例或事件模型泄漏到组件公共 API 中。

备选方案：向外暴露拖动事件和底层句柄。这会提升灵活性，但会把第三方依赖耦合到公共接口，增加未来替换成本，因此本次仅保留必要的内部封装。

### 5. 动态子区域变化采用“保守重算”策略

当 `children` 数量变化时，组件重新计算区域列表；如果只是删除区域，则保留现存区域的相对顺序，并对比例做归一化；如果新增区域，则回退到均分或按现存比例扩展。这样可以确保 demo 中“子区域动态销毁后布局变化”行为稳定且可预期，避免旧分割线索引与新子节点错位。

备选方案：尝试通过 key 精细继承每个区域历史尺寸。该方案更强大，但需要额外定义区域身份模型，本次先采用更稳妥的保守重算策略。

### 6. 开发示例与测试围绕“组合 + 交互”两条主线构建

开发示例覆盖横向、纵向、嵌套和动态删除场景；测试则聚焦区域数量与方向 class、分割线是否按配置出现、子区域更新后结构是否正确。拖动行为优先验证状态更新与结构稳定性，不依赖过重的端到端可视化断言。

备选方案：仅提供静态渲染示例。这样无法证明拖动和动态销毁的关键需求已被覆盖，因此不采用。

## Risks / Trade-offs

- [拖动库与 React 生命周期耦合] → 使用 `useEffect`/`useLayoutEffect` 统一注册与销毁拖动实例，确保子区域变化时清理旧绑定
- [比例重算可能导致动态删除后布局跳变] → 采用相邻区域保序 + 总量归一化策略，并在 demo 中明确验证删除后的表现
- [嵌套场景下拖动命中区互相影响] → 将拖动绑定限定在当前层级的分割线元素上，避免事件作用到子级 `CSplitArea`
- [缺少最小尺寸约束时可能出现极窄区域] → 先在内部为比例调整设置基础下限，后续如有需要再扩展正式 API
- [拖动测试在 jsdom 中能力有限] → 以逻辑层和结构层测试为主，把复杂交互验证留给 demo 与后续更高层测试

## Migration Plan

1. 在 `src/components/CSplitArea/` 下新增组件、样式与导出文件
2. 将 `CSplitArea` 接入 `src/components/index.ts` 与 `src/index.ts` 的导出链路
3. 在 `src/dev/` 中补充嵌套与动态删除示例，供人工验证
4. 在 `tests/` 中新增针对渲染结构和动态子区域变化的测试
5. 执行 lint、test、build 验证后发布

回滚策略：若实现引入问题，可通过移除 `CSplitArea` 导出与对应示例/测试回退，不影响现有组件 API。

## Open Questions

- `CSplitArea` 是否需要在首版公开最小区域尺寸相关属性，还是先以内置下限满足交互稳定性
- 是否需要为分割线暴露自定义 className / render hook，以满足未来更复杂的视觉定制
- 动态新增区域时采用“全部均分”还是“保留原有比例并给新增区域分配默认份额”，需要在实现前最终确认
