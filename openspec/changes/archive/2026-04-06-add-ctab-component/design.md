## Context

当前组件库已经形成了较稳定的组件组织方式：源码位于 `src/components/`，组件内部维护结构性 SCSS，主题外观由各主题样式表统一覆盖，导出通过 `src/components/index.ts` 与 `src/index.ts` 暴露。现有公开组件多采用 `C*` 前缀命名，并通过 `useTheme` / `mergeClasses` 注入主题类名与额外类名。

本次变更需要补齐标签页容器能力，提供 `CTab` 与 `CTabItem` 的组合式 API。与通过配置数组单独传入标题不同，提案要求 `CTab` 直接从子节点 `CTabItem.title` 中提取标签头，并渲染对应内容区域。该能力是新增公开 API，要求保持无破坏性，并且不引入新的运行时依赖。

## Goals / Non-Goals

**Goals:**
- 提供声明式的 `CTab` / `CTabItem` 组合能力，让使用方直接以父子结构组织标签头与面板内容。
- 由 `CTab` 自动读取 `CTabItem.title` 渲染标签标题，减少外部重复维护标题数组与内容映射。
- 保持与现有组件库一致的主题接入、导出方式、样式命名和测试策略。
- 以最小可用 API 完成首版能力，保证易用且便于后续扩展。

**Non-Goals:**
- 首版不引入新的外部依赖或复杂状态管理方案。
- 首版不覆盖可关闭标签、拖拽排序、异步懒加载等高级标签页能力。
- 首版不强行设计完整受控/非受控双模式 API，避免在需求尚未明确时扩张接口面。

## Decisions

### 1. 使用组合式子组件协议，而不是配置数组

`CTab` 将基于 `React.Children.toArray` 遍历子节点，只将合法的 `CTabItem` 元素视为参与渲染的标签项。`CTabItem` 作为声明式载体，负责提供 `title` 和内容；真正的标签头列表与面板 DOM 由 `CTab` 统一输出。

这样可以直接满足提案中的“标题从子项读取”的要求，也让内容与标题天然保持邻近，避免使用方维护两份数据。相比“`tabs` 数组 + children 插槽”或“标题数组单独传入”的方案，这种方式更符合当前组件库的组合式使用习惯，心智负担更低。

### 2. 首版采用最小非受控激活状态

`CTab` 内部维护当前激活项，默认选中第一个可渲染的 `CTabItem`。内部标识优先复用 React child `key`，如果未提供则回退到按顺序生成的稳定 id，用于 `aria-controls` / `aria-labelledby` 关联以及激活项切换。

该设计能在不扩展额外公共 API 的前提下完成首版能力，避免过早引入 `activeKey` / `onChange` 等受控接口。相比一开始就同时支持受控与非受控模式，这种做法更容易实现、测试和稳定交付；如果后续规格明确需要受控模式，可在兼容的前提下增量加入。

### 3. 使用语义化 Tab 结构和基础可访问性标记

组件将输出统一的结构类名，例如：
- 根节点：`.cm-ctab`
- 标签头容器：`.cm-ctab__list`
- 标签按钮：`.cm-ctab__tab`
- 当前激活态：`.cm-ctab__tab--active`
- 内容面板：`.cm-ctab__panel`

标签头使用 `button[type="button"]` 渲染，并补充 `role="tablist"`、`role="tab"`、`role="tabpanel"`、`aria-selected`、`aria-controls`、`aria-labelledby` 等属性，保证基础语义和屏幕阅读器可识别性。首版优先覆盖点击切换；更复杂的键盘 roving tabindex、方向键循环等交互暂不纳入本次设计范围。

### 4. 结构样式与主题外观继续分层

组件级 `src/components/Tab/index.scss` 只承载布局、结构和状态类名约定；主题相关的视觉细节（边框、背景、激活态色彩、阴影等）继续落到各主题样式文件中，以 `.cm-ctab*` 选择器增量补充。这与当前 `Menu` 等组件的组织方式一致，能保持主题系统的统一性。

`CTab` 将复用现有的 `useTheme` 与 `mergeClasses` 模式，支持 `className`、`theme`、`data-testid` 等与现有组件一致的扩展入口，避免引入新的主题接线方式。

### 5. 目录、导出与测试按现有库约定落位

实现文件放在 `src/components/Tab/` 目录下，并通过 `src/components/index.ts` 与 `src/index.ts` 暴露公共导出。测试新增到 `tests/`，重点覆盖：
- 能从 `CTabItem.title` 生成标签头
- 默认展示首个标签内容
- 点击标签后切换内容面板
- 导出后的组件可被消费者直接使用

相比把测试散落到开发示例或仅依赖人工预览，RTL 测试能更稳定地验证组合式 API 与交互行为。

## Risks / Trade-offs

- [子项未设置 React key 时，重排可能导致激活项回退] → 优先使用 child `key` 作为内部标识；未提供 key 时接受按顺序回退，并在文档中建议为动态标签提供稳定 key。
- [`CTabItem` 主要承担声明作用，自身不直接输出完整 DOM] → 在组件说明和示例中明确其“由 `CTab` 读取并渲染”的使用模式，避免用户误把它当成独立可视组件。
- [新增标签组件需要同时补齐多套主题样式] → 先定义稳定的 `.cm-ctab*` 结构类契约，再分别在 default / win98 / winxp 主题中实现视觉覆盖，降低样式漂移风险。
- [首版不提供受控 API，灵活性有限] → 先交付最小必需能力；若后续出现外部联动切换场景，再通过兼容方式补充受控接口。

## Migration Plan

1. 在 `src/components/Tab/` 新增 `CTab`、`CTabItem` 及其结构样式。
2. 在主题样式中补充 `.cm-ctab*` 外观定义，确保默认主题和现有系统主题表现一致。
3. 更新 `src/components/index.ts` 与 `src/index.ts`，将新组件纳入公开导出。
4. 增加对应测试用例，验证标题提取、默认激活与切换行为。

该变更为纯新增能力，不涉及数据迁移。若上线后需要回滚，只需移除新增组件导出与对应样式、测试即可，不会影响现有组件消费者的既有行为。

## Open Questions

当前没有阻塞本次设计落地的开放问题。若后续规格明确要求支持方向键导航、`Home` / `End` 快捷切换或受控模式，可作为后续增量能力继续扩展，而不影响本次首版的基础结构设计。
