## Context

当前开发入口 `src/dev/main.tsx` 直接渲染 `<Win98Theme />`，主题选择被硬编码在入口层，无法在不改入口代码的前提下切换到其他主题。`CScreenManager` 当前仅做 children 透传，而 `CWindowManager` 已具备注册、去重、按构造器渲染的管理能力，二者职责不对齐。与此同时，现有代码中未提供 `WinXpTheme`，且存在与主题上下文相关的残留测试实现，需要在本次变更中统一语义并清理不一致测试。

约束与前提：
- `CScreen` 保持在 theme 内声明（由主题决定屏幕结构）。
- 主题切换采用代码常量，不引入运行时配置中心或外部依赖。
- 保持 React + TypeScript 现有类组件风格与导出组织方式。

## Goals / Non-Goals

**Goals:**
- 让开发入口从“固定 Win98”升级为“主题切换挂载点”。
- 在不改变 `CScreen` 声明位置的前提下，让 `CScreenManager` 具备可验证的屏幕管理职责（注册/去重/渲染）。
- 新增 `WinXpTheme` 并可通过常量在 Win98/WinXp 之间切换。
- 清理与当前实现不一致的残留测试，确保测试覆盖与行为契约一致。

**Non-Goals:**
- 不引入动态主题持久化（localStorage、URL、服务端配置等）。
- 不在本次变更中实现跨屏拖动或复杂屏幕编排能力。
- 不强制引入全局 ThemeContext（本次仅要求入口常量切换可用）。

## Decisions

### 1) 入口主题切换组件化，替代直接引入 `Win98Theme`
- 决策：在 `main.tsx` 使用一个明确的主题切换组件（或同等语义封装），由常量 `ACTIVE_THEME` 决定挂载 `Win98Theme` 或 `WinXpTheme`。
- 原因：将“入口挂载”与“具体主题实现”解耦，减少后续新增主题时对入口代码的侵入。
- 备选方案：
  - 继续在 `main.tsx` 直接写条件表达式：可行，但可读性与职责边界更弱。
  - 引入完整 ThemeContext：扩展性更强，但超出本次“常量切换”的目标。

### 2) `CScreenManager` 对齐 `CWindowManager` 的管理模型
- 决策：`CScreenManager` 采用与 `CWindowManager` 一致的“合法类型识别 + 注册 + 去重 + 渲染”模式，但管理对象改为 `CScreen`。
- 原因：复用团队已接受的管理器行为模型，降低认知成本并让测试可对齐。
- 备选方案：
  - 保持透传：实现成本低，但无法体现 manager 语义，后续扩展困难。
  - 将 `CScreen` 移出 theme 由 manager 注入：会破坏本次已确认的“screen 在 theme 内声明”前提。

### 3) 基类抽象采用“延迟抽象”策略
- 决策：本次先评估并尽量提取共享逻辑；仅在两类 manager 共享行为与类型边界清晰时抽取轻量基类（如内部通用注册逻辑）。若类型约束差异过大，则保留两个具体 manager，避免过抽象。
- 原因：当前 `Window` 与 `Screen` 的类型体系不完全一致，强行统一可能引入额外复杂度与泛型负担。
- 备选方案：
  - 立即强制抽象 `BaseManager<T>`：代码看似统一，但可能牺牲可读性并增加类型噪声。
  - 完全不评估抽象：与变更目标不符，无法回答“可否抽基类”的设计问题。

### 4) 残留测试按“行为契约一致性”清理
- 决策：移除与当前主题机制不一致、且无对应实现支撑的测试；保留并补齐与 `screen-class-registration`、`dev-theme-selection` 契约直接相关的测试。
- 原因：测试应反映当前契约而非历史草案，避免 CI 中出现“假失败/假通过”。
- 备选方案：
  - 全量保留历史测试：会持续制造噪音并阻碍迭代。
  - 全量删除主题相关测试：风险过高，缺失回归保障。

## Risks / Trade-offs

- [Risk] `CScreenManager` 行为变更可能影响现有渲染顺序或 key 稳定性 → Mitigation: 为注册、去重、无效元素过滤、渲染数量增加针对性单测。
- [Risk] `WinXpTheme` 视觉/结构与 Win98 差异导致测试脆弱 → Mitigation: 测试聚焦结构契约与可选择性，不绑定过细样式细节。
- [Risk] 基类抽象边界不清引发类型复杂化 → Mitigation: 以“可读性优先”作为抽象门槛，不满足门槛则维持具体实现。
- [Risk] 残留测试识别不全导致后续回归盲区 → Mitigation: 在任务阶段明确“移除原因 + 替代测试”配对关系。

## Migration Plan

1. 先完成主题切换入口与 `WinXpTheme` 的最小可用接入，保持 `CScreen` 仍由 theme 声明。
2. 将 `CScreenManager` 升级为可管理 `CScreen` 的实现，并补齐对应单测。
3. 按评估结果决定是否抽取轻量共享基类/公共逻辑；若抽取，保证不改变对外 API。
4. 清理残留测试并新增替代测试，确保 CI 覆盖新契约。
5. 验证 lint/test/build 全通过后进入实现阶段。

回滚策略：若 manager 升级引发回归，可先回退到透传实现并保留主题切换能力，待问题定位后再逐步恢复 manager 能力。

## Open Questions

- 若不抽象完整基类，是否接受“提取共享私有工具函数”作为本次“可抽象评估”的落地结果？
