## Context

当前 Window 组件在 interactionMode 为 static 时，拖拽或缩放会显示虚线外框作为过程反馈，但外框尺寸与最终窗口尺寸存在偏差。该行为影响用户对拖拽结果的预期一致性，需要在不改变 follow 模式交互的前提下修复。

## Goals / Non-Goals

**Goals:**
- static 模式下拖拽/缩放的虚线外框尺寸与窗口最终尺寸保持一致。
- 交互模式差异只体现在更新时机，不影响外框与最终尺寸的数值一致性。

**Non-Goals:**
- 不调整 Window 组件的视觉样式或主题令牌。
- 不更换拖拽库或引入新依赖。

## Decisions

- 复用现有 Window 组件的尺寸计算逻辑，确保外框尺寸来源于同一套尺寸/位置计算结果。
  - 备选方案：为 static 模式单独维护外框尺寸状态。否决原因：双路径易造成漂移与维护成本上升。
- 将外框渲染尺寸与最终状态更新解耦，仅在渲染层应用“即时尺寸”，而最终状态仍按 static 模式在交互结束时提交。
  - 备选方案：在 static 模式下完全禁用拖拽过程反馈。否决原因：用户体验退化。

## Risks / Trade-offs

- 外框与最终尺寸来源统一后，若计算过程依赖拖拽库的中间值，可能放大边界情况下的抖动 → 通过对齐现有 follow 模式的计算路径并复用其约束逻辑降低风险。
- 调整渲染层尺寸可能影响测试快照或交互断言 → 需要补充/更新 static 模式下的拖拽/缩放断言。
