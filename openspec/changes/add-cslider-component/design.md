## Context

当前组件库缺少统一的滑块输入组件，业务侧如果需要连续或离散数值选择，只能自行组合轨道、滑块按钮和拖拽逻辑。此次变更目标是在现有 React 18 + TypeScript 组件库中补齐 `CSlider`，并保持与现有组件的组织方式一致：组件源码放在 `src/components/`，样式使用独立 `scss`，通过 `src/components/index.ts` 与 `src/index.ts` 统一导出，测试使用 Jest + React Testing Library。

仓库内已经在 `CSplitArea`、`WindowTitle`、`Widget`、`IconContainer` 等组件中使用 `@system-ui-js/multi-drag` 处理拖拽，因此 `CSlider` 不需要重复实现底层 pointer 状态管理。组件还需要遵循现有约定：

- 使用 `cm-` 前缀和 BEM 风格类名。
- 使用 `mergeClasses`/主题包装组件合并主题类名与外部 `className`。
- 通过显式类型定义避免 `any`。
- 通过测试覆盖拖拽、步进对齐、边界值与局部样式入口。

## Goals / Non-Goals

**Goals:**

- 提供一个单值滑块组件，支持最小值、最大值与步进约束。
- 复用 `@system-ui-js/multi-drag` 实现滑块按钮拖动与位置更新。
- 提供关键子元素的 className 定制入口，至少覆盖根节点、轨道、填充区和滑块按钮。
- 保持与现有组件相同的导出、样式和测试组织模式，降低维护成本。
- 在拖拽与点击更新值时保持值计算规则一致，避免 UI 位置与回调值不一致。

**Non-Goals:**

- 不在本次变更中支持多滑块范围选择（双滑块/区间选择）。
- 不引入新的样式方案或主题系统扩展。
- 不在本次变更中处理复杂标记系统（刻度文案、提示气泡、格式化标签）。
- 不修改 `@system-ui-js/multi-drag` 的实现或组件库现有拖拽基础设施。

## Decisions

### 1. 使用单滑块、单一数值模型

`CSlider` 首期聚焦单值选择，内部始终围绕“当前值 ↔ 百分比位置”做映射，不扩展到多手柄或区间模型。

- 选择该方案的原因：与提案需求完全对齐，状态模型简单，拖拽边界、步进对齐和样式定制都更容易验证。
- 备选方案：直接设计为多手柄通用模型。
- 不采用备选方案的原因：会显著增加碰撞处理、焦点管理和 API 复杂度，超出本次变更范围。

### 2. 统一值换算管线，所有交互共用同一套约束逻辑

组件内部采用统一的值换算流程：根据轨道尺寸将 pointer 位置转换为比例值，再映射到 `[min, max]` 范围，最后通过 `step` 做对齐并截断到边界。拖拽、轨道点击以及外部值同步都复用同一套换算规则。

- 选择该方案的原因：可以避免不同入口各自计算导致的边界不一致问题，也便于测试步进舍入与边界裁剪。
- 备选方案：拖拽、点击、受控更新分别处理各自的值计算。
- 不采用备选方案的原因：容易出现视觉位置正确但回调值不同步的问题，后续维护成本更高。

### 3. 复用 `@system-ui-js/multi-drag` 管理拖拽生命周期

组件通过 `@system-ui-js/multi-drag` 绑定滑块按钮，在拖拽过程中仅关注“当前位置如何换算为值”，并在组件卸载或依赖变化时按现有模式清理拖拽实例。

- 选择该方案的原因：仓库内已有成熟使用方式，能复用 pointer 事件处理、禁用与清理约定，减少底层交互 bug。
- 备选方案：直接使用原生 pointer 事件自行实现拖拽。
- 不采用备选方案的原因：会重复实现已有能力，增加事件清理、触摸兼容和测试 mock 成本。

### 4. 提供结构化的局部 className 定制入口

除组件根节点 `className` 外，`CSlider` 额外提供面向关键子元素的 className 配置对象，用于分别定制轨道、填充区和滑块按钮。组件内部仍保留稳定的默认 BEM 类名，外部类名只做附加，不替换内部语义结构。

- 选择该方案的原因：既满足提案中的局部样式定制诉求，也能保持组件内部 DOM 结构和测试选择器稳定。
- 备选方案：仅暴露单个根节点 `className`，由使用方覆写所有后代样式。
- 不采用备选方案的原因：调用方需要依赖内部 DOM 层级，样式耦合高，后续结构调整风险更大。

### 5. 保持现有组件组织与导出方式

组件目录采用与现有组件一致的结构：`src/components/CSlider/CSlider.tsx`、`src/components/CSlider/index.ts`、`src/components/CSlider/index.scss`，并在 `src/components/index.ts` 与 `src/index.ts` 中统一导出组件和类型。

- 选择该方案的原因：与现有组件库结构一致，便于消费者按统一入口使用，也方便后续补充文档和测试。
- 备选方案：只在顶层入口直接导出组件文件。
- 不采用备选方案的原因：会破坏现有 barrel 导出模式，不利于组件目录内聚。

### 6. 以行为测试覆盖值计算与拖拽集成

测试层面沿用 `CSplitArea` 的 mock 模式模拟 `@system-ui-js/multi-drag`，重点验证初始渲染、拖拽更新、步进对齐、边界裁剪、局部 className 注入与导出可用性。

- 选择该方案的原因：可以在不依赖真实浏览器拖拽的前提下稳定覆盖关键交互。
- 备选方案：仅做静态渲染测试或端到端验证。
- 不采用备选方案的原因：前者无法证明交互正确，后者对当前组件库来说过重且反馈慢。

## Risks / Trade-offs

- [轨道尺寸依赖真实布局] → 在测试中显式 mock `getBoundingClientRect`，并将值换算逻辑尽量收敛为纯函数或可独立验证的辅助逻辑。
- [步进舍入导致视觉位置与预期不符] → 明确采用统一的舍入规则，并在临界值（最小值、最大值、半步附近）增加测试用例。
- [局部 className 入口过多导致 API 膨胀] → 仅暴露稳定且必要的关键节点，不开放与内部实现强绑定的细粒度样式钩子。
- [拖拽实例生命周期处理不当可能造成泄漏] → 复用仓库中已验证的 `multi-drag` 初始化与清理模式，在依赖变化和卸载时统一销毁或禁用。
- [点击轨道与拖拽按钮行为不一致] → 两种交互都走同一套坐标转值逻辑，只在触发入口上做区分。

## Migration Plan

1. 新增 `CSlider` 组件目录、样式文件和类型导出。
2. 在组件索引与库入口中暴露 `CSlider` 及其相关类型。
3. 补充针对拖拽与值换算的测试，确认现有组件不受影响。
4. 通过 lint、test、build 验证后随组件库版本一并发布。
5. 若上线后发现交互或样式问题，可通过移除顶层导出并回退组件目录实现完成回滚；该变更不涉及数据迁移。
